How to Find Your Certificate's Thumbprint

  1. Press Win + R, type certlm.msc, and hit Enter.
  2. Go to Personal > Certificates and find your certificate.
  3. View the Thumbprint
  • Double-click your certificate.
  • Go to the Details tab.
  • Scroll down and look for Thumbprint.
  • Copy the long string of letters/numbers—this is your thumbprint

Create a GUID (appid):

  1. Open PowerShell as Administrator
  2. Run [guid]::NewGuid()

Now it should appear in the drop down menu. Select it and hit OK

To double check if it’s correct, open certlm and look in Personal and Trusted Root Certification Authorities. If it appears in both places then you’re good to go.

Fájlrendszerek

Ext4 → sudo mkfs.ext4 /dev/sdb1 → Standard Linux file system, supports journaling and large volumes.

XFS → sudo mkfs.xfs /dev/sdb1 → High-performance, great for large files.

Btrfs  → sudo mkfs.btrfs /dev/sdb1 → Advanced features like snapshots and compression.

FAT32 → sudo mkfs.vfat -F 32 /dev/sdb1 → Good for USB drives, cross-compatible, but outdated.

NTFS → sudo mkfs.ntfs /dev/sdb1 → Windows default file system, supports permissions and compression.

exFAT → sudo mkfs.exfat /dev/sdb1 → Optimized for flash storage, works across platforms.

Links

Soft links, or symbolic links, act as shortcuts that reference the original file or directory rather than storing the actual data. If the target file is removed, the symbolic link becomes broken because it depends on the original path.

Hard links, on the other hand, reference the same inode as the original file, meaning they contain the actual data rather than just a path. Changes made to a hard link affect all linked copies because they are fundamentally the same file. If the original is deleted, the hard link remains intact since the data itself is still present on the disk. Hard links work only within the same filesystem and cannot link to directories, while soft links can cross filesystems and work for both files and directories.

Soft Links

Soft links are like shortcuts—they point to the original file, but break if the file is deleted.

Hard Links

Hard links, on the other hand, reference the actual data on disk, so they reflect changes both ways and persist even if the original file is removed.

Fájlhozzáférések

chown xxx /path/to/file

7 = Read (4) + Write (2) + Execute (1) → rwx

6 = Read (4) + Write (2) → rw-

5 = Read (4) + Execute (1) → r-x

4 = Read (4) → r--

3 = Write (2) + Execute (1) → wx

2 = Write (2) → w--

1 = Execute (1) → x--

0 = No permissions → ---

chmod user:group /path/to/file

-R → Works on both commands, applies their effect to the directory and all files and subdirectories within it.

ACL-ek

getfacl /path/to/file → Retrieves the ACL settings of a file or directory

setfacl → Modifies ACL rules

setfacl -m u:username:rwx file.txt → Assign ACL permissions to a user

setfacl -m g:groupname:rwx file.txt → Set permissions for a group

setfacl -m m:rwx file.txt → Configure the mask, limiting the maximum permissions for non-owner entries,

setfacl -x u:username file.txt → To remove a specific ACL entry

setfacl -m d:u:username:rwx /path/to/directory → Setting default ACLs applies permissions to all newly created files within a directory

setfacl -b file.txt → To clear all ACLs on a file, use

Examples:

Without ACLs:

setfacl -m u::rw /home/basd/acltest/

setfacl -m u:basd:rw /home/basd/acltest/

setfacl -m m:rw /home/basd/acltest/

After creating a subfolder

setfacl -m d:u:basd:rw /home/basd/acltest/

After creating another subfolder, the default ACL applies to it. It does not change the existing folder’s permissions.

To change permissions on the existing folder

setfacl -R -m u:basd:rw /home/basd/acltest/

And to limit permissions to read only except the owner and group ones

setfacl -R -m d:m:r /home/basd/acltest/

setfacl -R -m m:r /home/basd/acltest/

Because basd is the owner of subfolder, it still has rwx permissions, but it isn't the owner of subfolder2, and has only read access.

Tűzfal, proxy

If you allow traffic from a specific IP to a specific port, other sources trying to use that port are denied unless explicitly allowed. This restriction applies only to that port, while other ports remain unaffected.

Enable UFW

ufw enable

Deny incoming and allow outgoing by default

ufw default deny incoming

ufw default allow outgoing

Allow specific ports on tcp or udp

ufw allow 80/tcp

ufw allow 443/tcp

ufw allow 67/udp

ufw allow 68/udp

Reset UFW (Clear all rules)

ufw reset

Limit SSH to one source and one destination address while preventing brute force attacks

ufw deny 22/tcp

ufw limit from 192.168.1.1 to 192.168.1.100 port 22 proto tcp

Template

ufw [ allow / deny / limit ] ( [ in / out ] on [ interface ] ) [ from <source IP> ] [ to <destination IP> ] [ port <port number> ] [ proto <protocol> ]

Examples

ufw deny from 192.168.1.1                                        # Deny traffic from the specified host

ufw allow from 192.168.1.0/24 to any port 80 proto tcp        # Allow 80/tcp traffic from the subnet to any destination

ufw allow in on eth0 to any port 22 proto tcp                # Allows 22/tcp in on eth0 but doesn’t block other ports or 22/tcp on other interfaces.

ufw delete allow 22/tcp                                        # remove the rule

Configure Dovecot

sudo nano /etc/dovecot/conf.d/10-mail.conf

Test the mail server

echo "Test email" | mail -s "Test Subject" user@homelab.com

Log in as the recipient and view the contents of ~/Maildir/new/. The email should appear there.
